# Poster Export Handover And Next-Step Spec

## Objective

Produce print-ready PNG exports for the two HTML posters:

- `poster_v2.html`
- `poster_dhq.html`

Each final export must satisfy **all** of the following:

1. Exact **A0 landscape** canvas size.
2. Final PNG metadata tagged at **300 DPI**.
3. A small white printer-safe margin around the poster content.
4. No rendering artefacts, especially in the footer or bottom-right corner.
5. No reduction in true visual fidelity caused by scaling a smaller raster source upward.

This last point is critical:

- A file can be exact A0 and tagged `300 DPI`, while still being **lower quality than true 300 DPI** if the source image was smaller and then upscaled.
- That is **not acceptable** for final delivery.

## Non-Ambiguous Definitions

These terms must be kept separate.

### 1. Final Canvas Resolution

This is the pixel size of the final PNG file.

For A0 landscape at 300 DPI:

- width: `14043 px`
- height: `9933 px`

If a PNG is this size, it has the correct **canvas resolution** for A0 at 300 DPI.

### 2. Metadata DPI

This is the DPI value embedded in the PNG metadata.

Required:

- `300 x 300 DPI`

This is necessary, but **not sufficient** for quality.

### 3. Source Detail Resolution

This is the actual amount of detail in the raster image **before** it is placed into the final A0 canvas.

For the poster content area to be true 300 DPI, the source raster used for the poster itself must be at least as large as the final placed poster area.

If the source poster raster is smaller and then scaled up, detail is lost:

- fonts soften
- thin lines soften
- screenshot/UI details soften

This is true even if the final PNG still says `300 DPI`.

### 4. Printer Margin

This is the white space intentionally added around the poster because the printer cannot print fully edge-to-edge.

Current working margin target:

- `10 mm`

At 300 DPI this is approximately:

- `118 px`

## What Was Done So Far

### Initial goal

The task was to export the HTML posters as A0 PNGs with a small white printer-safe margin.

### First attempt

An A0-margin PNG was generated by:

1. taking an existing large PNG render of the poster
2. placing it on an exact A0 white canvas
3. tagging the result as `300 DPI`

This produced files with correct final pixel dimensions and metadata, but a white footer artefact appeared in the bottom-right area.

### Investigation

The following was observed:

- The HTML in the browser looked correct.
- The artefact did not consistently appear in ordinary browser viewing.
- The artefact appeared in some high-scale export paths.
- The issue was therefore traced primarily to the **export/render path**, not to the visual HTML as seen in the browser.

### Footer hardening changes

Small footer layout hardening changes were made in:

- `poster_v2.html`
- `_render_poster_v2.html`
- `poster_dhq.html`
- `_render_poster_dhq.html`

These changes improved robustness, but they were **not** the main reason the artefact occurred.

### Alternate render tests

Several rendering methods were tested:

1. Chrome headless screenshot at very large scale.
2. Wrapper HTML / iframe-based render.
3. Chrome PDF export, then rasterize PDF to PNG.
4. Puppeteer browser element capture at multiple device scale factors.

Findings:

- Some export methods introduced white footer artefacts.
- Some export methods clipped or dropped the footer.
- Some export methods produced a visually correct poster but at too small a raster size to preserve true 300 DPI detail.

### Important compromise that occurred

A clean browser capture was found at a stable scale factor, but it was smaller than the final placed poster area.

That clean source was:

- `11890 x 8410`

The final placed poster area inside the A0 canvas was:

- `13710 x 9697`

That means the clean source had to be scaled up by about `15.3%`.

This gave:

- correct final A0 canvas size
- correct `300 DPI` metadata
- but **not true native 300 DPI source detail**

Approximate effective source-detail DPI:

- width: `11890 / 13710 * 300 ≈ 260 DPI`
- height: `8410 / 9697 * 300 ≈ 260 DPI`

This is the key lesson:

- the file met the final output spec by dimensions
- but it failed the stricter quality requirement because it had been upscaled

That compromise is **not acceptable as the final definition of success**.

## Current State

### What is currently true

- The current A0 files have the correct A0 canvas pixel size:
  - `14043 x 9933`
- The current A0 files are tagged:
  - `300 x 300 DPI`
- The obvious footer artefact was removed from the current rebuilt files.

### What is not acceptable about the current state

The current rebuilt A0 files were derived from a source raster that was too small and had to be enlarged.

Therefore the current files must be treated as:

- visually improved
- dimensionally correct
- metadata-correct
- but **not final-quality true-detail 300 DPI exports**

## Explicit Acceptance Criteria For Final Delivery

The final export is acceptable only if **all** of these are true.

### Geometry and metadata

1. Final file size is exactly:
   - `14043 x 9933`
2. PNG metadata is exactly:
   - `300 x 300 DPI`
3. Margin exists and is intentional:
   - target `10 mm` minimum

### Fidelity

4. The poster raster used before compositing to A0 is **not smaller** than the final placed poster area.
5. Preferably, the source raster is slightly larger than the placed poster area, so the final composite does not require enlargement.
6. No raster upscaling of the poster source is allowed unless explicitly approved.

### Visual integrity

7. No white rectangular artefacts or footer corruption.
8. Footer text fully visible and clean.
9. Screenshot/UI details remain crisp.
10. Small text remains as sharp as possible for large-format print.

## Required Next Step

The next step should be a **true-detail export path** that avoids the buggy single-pass high-scale capture.

### Recommended approach: tiled browser capture and stitching

Render the poster in the browser at a stable scale factor that does **not** corrupt the footer, then capture it in multiple tiles and stitch the tiles together.

This allows:

- browser-faithful rendering
- no footer artefacts
- source raster larger than the final placed poster area
- no unacceptable upscaling

### Minimum requirement for the stitched poster source

For each poster, the stitched source image used before A0 compositing must be at least:

- `13710 x 9697`

Anything smaller than that means the final A0 poster area would be upscaled and would fail the true-detail requirement.

### Better target

Aim for a stitched source somewhat larger than the final placed poster area, for example around:

- `15000+ px` width
- matching height proportionally

Then scale down slightly into the A0 content area if needed.

Scaling down is acceptable.
Scaling up is not acceptable.

## Proposed Exact Workflow

### Phase 1: capture a clean source raster

1. Open the HTML poster in Chrome/Puppeteer.
2. Use a stable device scale factor that does not introduce footer artefacts.
3. Capture the poster as overlapping tiles.
4. Stitch tiles into one poster image.
5. Verify the stitched image contains the full footer and no white artefacts.

### Phase 2: validate source-detail size

6. Measure the stitched poster source dimensions.
7. Confirm source width and height are both at least as large as the final placed poster area.
8. If not, increase capture scale or tile count and repeat.

### Phase 3: build final print file

9. Create a white A0 landscape canvas:
   - `14043 x 9933`
10. Reserve `10 mm` safe margin:
   - about `118 px`
11. Place the stitched poster centered within the printable area.
12. Do not enlarge the poster source during this step.
13. Save PNG with metadata:
   - `300 x 300 DPI`

### Phase 4: final QA

14. Visually inspect:
   - footer
   - bottom-right corner
   - screenshot details
   - smallest text
15. Confirm no white artefacts.
16. Confirm exact pixel size and DPI metadata.

## What Must Not Happen Again

The following must be treated as failure conditions unless explicitly approved by the user:

1. Delivering a file that is exact A0 and tagged 300 DPI, but built from a smaller source raster that was enlarged.
2. Calling the output "300 DPI quality" when only the canvas/metadata meets 300 DPI.
3. Treating browser-visible correctness as proof that the export path is also correct.
4. Patching around artefacts in the final raster without first verifying whether the export method itself is defective.

## Short Summary

The real objective is **not** merely:

- "make a PNG that is A0-sized and tagged 300 DPI"

The real objective is:

- make a PNG that is A0-sized
- tagged 300 DPI
- has a printer-safe margin
- has no export artefacts
- and preserves **true source-detail sharpness consistent with 300 DPI large-format print**

The current files satisfy the geometry and metadata requirements, but they do **not yet satisfy the no-upscaling / true-detail fidelity requirement**.

The next correct action is to produce a tiled, stitched, browser-faithful high-resolution source raster and rebuild the final A0 files from that.
